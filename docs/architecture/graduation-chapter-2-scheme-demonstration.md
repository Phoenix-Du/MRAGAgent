# 2 方案论证

本章围绕“面向通用问答与文搜图场景的多模态 RAG 原型系统”进行方案论证。第一章已经从检索增强生成、多模态检索、网页解析与视觉语言模型等方向说明了课题背景和研究现状，本章进一步回答“为什么采用当前技术路线”的问题。系统并不是单一模型调用脚本，而是一个需要同时处理自然语言请求、开放网页、图片候选、表格结构、上下文记忆、外部模型服务和前端交互的综合工程系统。因此，本章从系统需求、可行性、开发工具与开源组件选型、关键技术以及基本方案制定等方面展开分析，为第三章系统设计和第四章系统实现提供依据。

## 2.1 引言

随着大语言模型和多模态模型的发展，问答系统已经不再局限于依赖模型参数知识直接生成答案。对于真实网页场景而言，用户提出的问题往往具有时效性、开放性和多模态特征：一方面，通用问答需要从网页、文档和表格等外部证据中获取可靠知识；另一方面，文搜图任务不仅要根据文本描述召回图片，还要判断图片是否真正满足主体、场景、风格和空间关系等约束。若仅采用传统文本 RAG，系统难以利用网页中的图片、表格和结构信息；若仅调用图片搜索接口，系统又难以保证图片链接可访问、内容相关且能够被解释。

基于上述问题，本课题确定的系统建设目标是：设计并实现一个可运行、可测试、可扩展的多模态 RAG 原型系统，将通用问答和文搜图两类任务组织到统一服务入口下，通过任务规划、证据构建、多模态检索、视觉语言模型回答、安全控制和运行观测等环节形成完整链路。本章的重点不是展开具体模块代码，而是在设计实现之前明确需求边界、技术路线和组件选型依据。

## 2.2 系统需求分析

系统面向开放域信息获取场景，核心用户可以通过浏览器前端或 API 提交自然语言请求。系统需要根据请求内容自动判断任务类型，并组织对应的检索、解析和回答链路。与普通问答系统相比，本系统的需求具有两个特点：其一，输入和证据来源多样，可能包括用户 query、指定 URL、用户上传文档、搜索结果网页和开放域图片；其二，回答过程较长，需要多个外部服务和本地模块协同，因此必须具备安全边界、降级能力和可观测能力。

| 功能类别 | 功能需求 | 说明 |
|---|---|---|
| 统一交互入口 | 提供聊天式 Web 页面和 `/v1/chat/query` API | 用户无需区分网页问答、RAG 问答和图片搜索接口，系统内部根据意图分流 |

| 图文匹配与视觉回答 | 使用 Chinese-CLIP 粗排图片，并在最终回答阶段使用 VLM 排序、过滤和解释 | 对复杂空间关系等场景使用 VLM 进行严格判断 |
| 澄清对话 | 对信息不足的请求进行追问 | 例如天气问题缺少城市，或图片搜索请求缺少明确主体 |
| 会话记忆与偏好 | 保存用户上下文、偏好和 pending 澄清状态 | 支持后续多轮补全和回答风格控制 |
| 前端展示 | 展示回答、证据、图片、进度和运行标记 | 用户可以看到系统执行阶段和结果来源 |

其中，通用问答链路的核心需求是“证据优先”。系统不能只把用户问题交给大模型直接回答，而应优先组织外部证据。对于指定 URL，系统需要直接采集该网页；对于没有指定材料的开放问题，系统需要先搜索候选网页，再通过相关性筛选和网页采集构造证据，最后送入 RAG 链路生成答案。

文搜图链路的核心需求是“可用图片证据优先”。开放域图片搜索结果中常见不可访问链接、防盗链、内容类型错误、缩略图与原图不一致等问题。如果直接把搜索引擎返回的 URL 交给后续模型和前端，会导致大量失败。因此系统必须在图片排序前进行可访问性验证和本地缓存，并将能够稳定读取的图片交给 Chinese-CLIP 和 VLM 使用。

## 2.3 系统可行性分析

从技术角度看，本系统具备实现可行性。首先，Python 生态已经提供了成熟的 Web 服务、异步 HTTP、数据建模、测试和模型调用工具。FastAPI 可以承担主编排服务和桥接服务的 HTTP 入口；Pydantic 可以对请求、响应和内部执行对象进行严格建模；httpx 支持异步外部服务调用；pytest 可以为接口、安全逻辑和核心分支提供回归测试。项目当前代码已经围绕这些工具形成了可运行结构。

其次，网页解析和多模态 RAG 可以通过开源系统集成实现。Crawl4AI 已经提供面向大模型的网页采集、浏览器渲染、HTML 清洗、Markdown 生成、媒体提取、表格识别和内容过滤等能力，本项目基于其网页采集能力进行统一封装和结构化转换。RAGAnything 支持文本、多模态内容、向量、实体和知识图谱等处理流程，本项目通过 RAGAnything Bridge 将内部证据文档适配为其多模态入库格式。也就是说，本课题的重点并不是从零实现浏览器爬虫或完整 RAG 引擎，而是在理解开源系统能力边界的基础上完成工程集成、数据适配和端到端编排。

再次，文搜图链路可以通过“搜索召回 + 可达性治理 + Chinese-CLIP 粗排 + VLM 最终回答”的方式实现。SerpAPI 等搜索服务可以提供开放域图片候选；Chinese-CLIP 能够对中文文本和图片进行跨模态相似度计算；Qwen-VL 或 OpenAI 兼容视觉语言模型可以在小规模候选上进行细粒度判断和自然语言解释。该组合既避免了完全依赖搜索标题的粗糙结果，也避免了对大量图片全部调用 VLM 带来的高成本和高延迟。

从经济角度看，系统主要使用开源框架和本地工程代码实现，基础开发和测试成本较低。搜索 API、VLM API 和部分 embedding 服务可能产生调用成本，但系统通过配置开关、fallback 路径和本地缓存降低了对真实外部服务的强依赖。在毕业设计阶段，可以用有限 API 调用完成典型链路验证，用自动化测试覆盖接口和安全逻辑。

从运行维护角度看，系统采用主服务、RAGAnything Bridge、Image Pipeline Bridge、Rasa Bridge 等分服务结构，既可以在本地通过脚本分别启动，也可以通过 Docker Compose 组合运行。主流程使用固定 API 契约连接各服务，外部服务失败时通过 runtime flags 显示降级状态，便于定位问题。整体来看，当前方案在技术、成本和维护上均具备可行性。

## 2.4 开发工具分析及选择

开发工具和开源组件的选择是本系统方案论证的重点。由于本课题是一个多模块协同的工程系统，选型不仅要考虑单个组件性能，还要考虑是否易于集成、是否支持异步调用、是否适合中文场景、是否便于测试和是否能够在外部服务缺失时降级。

### 2.4.1 开发语言与 Web 框架选择

系统后端选择 Python 作为主要开发语言，Web 框架选择 FastAPI。可选方案主要包括 Python + FastAPI、Java + Spring Boot、Node.js + Express/NestJS 等。

表 2-3 后端开发语言与 Web 框架选型比较

| 方案 | 优势 | 不足 | 适配结论 |
|---|---|---|---|
| Python + FastAPI | 与 LLM、RAG、爬虫、transformers、torch 等生态衔接最好；支持异步接口；Pydantic 数据建模天然集成 | 大规模企业治理能力弱于 Spring 体系 | 适合作为本系统主编排服务和桥接服务 |
| Java + Spring Boot | 工程治理、依赖管理和企业部署成熟 | 与多模态模型、Python SDK、深度学习组件集成成本较高 | 不适合作为本课题快速集成多模型原型的主技术栈 |
| Node.js + Express/NestJS | 前后端同语言，异步 I/O 能力强 | 深度学习和 RAG 相关 Python SDK 适配成本高 | 可用于轻量接口，但不适合作为核心模型编排层 |

本系统需要频繁调用 Crawl4AI、RAGAnything、transformers、torch、Chinese-CLIP、Rasa 以及 OpenAI 兼容模型 API。Python 在这些组件中的生态优势明显，因此选择 Python + FastAPI 更符合项目目标。FastAPI 的异步机制也适合处理搜索、网页抓取、RAG 查询和图片下载等 I/O 密集任务。

### 2.4.2 数据建模与接口约束工具选择

系统选择 Pydantic 作为请求、响应和内部执行数据结构的建模工具。可选方案包括普通 dict、dataclass、TypedDict 和 Pydantic。

普通 dict 使用灵活，但缺少字段边界和校验能力，容易在长链路中出现字段拼写错误或非法值传播。dataclass 适合内部轻量对象，但对 API 输入校验和 JSON schema 支持较弱。Pydantic 能够同时完成类型声明、默认值、范围约束、嵌套模型校验和 FastAPI 集成，因此更适合本系统。

当前系统将 `QueryRequest` 定位为外部 API 请求模型，禁止外部传入内部执行字段；将 `QueryExecutionContext` 作为后端内部执行载体，保存 `original_query`、`image_search_query`、`image_constraints` 和 `general_constraints` 等运行状态。该设计使外部接口更稳定，也避免用户绕过规划层直接注入内部检索 query 或约束对象。

### 2.4.3 网页采集与解析组件选择

网页采集可选方案主要包括 requests + BeautifulSoup、Scrapy、Playwright 自行封装、Firecrawl 和 Crawl4AI。

表 2-4 网页采集与解析方案比较

| 方案 | 优势 | 不足 | 适配结论 |
|---|---|---|---|
| requests + BeautifulSoup | 简单、轻量、易调试 | 难以处理 JavaScript 动态渲染，媒体、表格和结构化输出能力有限 | 适合作为弱 fallback，不适合作为主链路 |
| Scrapy | 爬虫框架成熟，适合批量站点采集 | 面向传统爬虫任务，对 LLM 友好 Markdown 和多模态输出支持不足 | 更适合离线爬虫，不适合本系统交互式 RAG 证据构建 |
| Playwright 自行封装 | 动态页面适配能力强，可控制浏览器行为 | 需要自行实现清洗、Markdown、媒体、表格和缓存策略 | 实现成本较高 |
| Firecrawl | 面向 LLM 的网页抽取能力较完整 | 更多体现为外部服务/产品化接口，部署和可控性受环境影响 | 可作为可行替代方案 |
| Crawl4AI | 支持浏览器渲染、HTML 清洗、Markdown 生成、媒体提取、表格识别、内容过滤和深度爬取策略 | 需要项目侧完成结构适配、异常处理和 RAG 入库转换 | 作为本系统网页智能采集能力的集成基础 |

本项目选择基于 Crawl4AI 的网页采集能力进行封装，主要原因是它更贴近“面向大模型的网页解析”需求。系统通过 `CrawlClient` 调用 Crawl4AI SDK 或远程服务，把网页采集结果转换为内部 `SourceDoc`，并尽量保留 Markdown、media、tables、links 和完整 `crawl4ai_full` 快照。这样做使网页不再只是纯文本段落，而是成为包含正文、表格、图片和链接的多模态证据对象。

### 2.4.4 RAG 框架与多模态知识引擎选择

RAG 方案可选类型包括自研简易向量库、LangChain/LlamaIndex 编排、RAGFlow、LightRAG 和 RAGAnything。

表 2-5 RAG 与多模态知识引擎方案比较

| 方案 | 优势 | 不足 | 适配结论 |
|---|---|---|---|
| 自研简易向量库 | 控制度高，便于小规模实验 | 需要自行实现切分、向量、检索、证据组织和多模态处理，工作量大 | 可作为本地 fallback，不适合作为主引擎 |
| LangChain/LlamaIndex | 生态成熟，连接器丰富 | 更偏应用编排框架，多模态知识结构和图谱能力需要额外组合 | 可用于快速原型，但不是本系统核心选择 |
| RAGFlow | 文档解析和知识库产品能力较完整 | 偏离线文档知识库，对本项目开放网页实时采集和文搜图链路适配成本较高 | 可作为参考方案 |
| LightRAG | 支持轻量知识图谱和混合检索 | 多模态内容处理需要额外封装 | 可作为底层能力参考 |
| RAGAnything | 面向多模态 RAG，支持文本、多模态内容分离、模态处理器、向量和图谱等流程 | 上游接口需要项目侧桥接，首次真实入库可能较慢 | 作为本项目多模态 RAG 引擎的集成对象 |

本项目选择 RAGAnything 的原因在于，它的能力范围与本课题对“网页多模态证据入库”的需求较匹配。网页采集结果中包含 Markdown、HTML、图片、表格和链接，若直接切分为文本段落，会丢失大量结构信息。RAGAnything 提供面向多模态内容的入库和查询能力，适合作为通用问答链路中的 RAG 引擎。本项目通过 `raganything_bridge.py` 将内部 `NormalizedDocument` 转换为 RAGAnything 可处理的 `content_list`，并保留远程图片物化、表格转换、弱证据 fallback 和 uid 级上下文回答等工程能力。

### 2.4.5 文本检索与重排模型选择

通用问答链路需要从搜索结果中选择更相关的网页，并在网页正文抓取后进一步过滤噪声。可选方案包括 BM25、普通 embedding 相似度、BGE Reranker 和 LLM 直接判断。

BM25 速度快、成本低，但对语义改写、同义表达和中文复杂问题的适配有限。普通 embedding 相似度适合大规模召回，但对候选之间的精细排序能力通常弱于 cross-encoder reranker。LLM 直接判断语义相关性效果可能较好，但成本高、延迟大，不适合对大量候选逐条调用。BGE Reranker 属于 cross-encoder 重排模型，能够同时读取 query 与候选文本，对中文语义匹配有较好适配性，适合在搜索召回后对较小候选集进行精排。

因此系统采用“搜索召回 + BGE 重排”的组合。搜索服务负责扩大候选范围，BGE 负责提高候选质量。该方案比仅使用搜索引擎排名更可控，也比对所有候选调用 LLM 更经济。

### 2.4.6 图像检索与跨模态匹配组件选择

文搜图链路可选方案包括仅使用图片搜索引擎、CLIP/Chinese-CLIP 进行图文匹配、VLM 全量判断，以及生成式文生图。由于本课题目标是“从开放域网页和图片结果中检索真实图片证据”，不是生成新图片，因此文生图不符合任务定位。

仅使用图片搜索引擎能够快速得到大量候选，但搜索结果质量受标题、网页上下文和搜索引擎排序影响，无法保证图片内容满足复杂约束。VLM 全量判断可以获得更细粒度理解，但对几十张候选逐张调用成本高、延迟大。CLIP 类模型适合对中等规模图片候选进行批量粗排，其中 Chinese-CLIP 对中文 query 更友好。

因此系统选择“SerpAPI 图片召回 + 可达性验证 + Chinese-CLIP 粗排 + 最终 VLM 判断”的方案。当前图像 pipeline 负责召回、URL 去重、可访问性验证、本地缓存和 Chinese-CLIP 过滤；最终 VLM 排序、空间关系严格过滤和答案生成集中在 `build_image_search_vlm_response()` 中完成。这样既避免 pipeline 和最终回答阶段重复 VLM 排序，也使复杂视觉判断发生在更小、更稳定的候选池上。

### 2.4.7 视觉语言模型选择

视觉语言模型可选方案包括 Qwen-VL、InternVL、LLaVA、GPT-4V/GPT-4o 系列以及纯 OCR/目标检测组合。纯 OCR 或目标检测适合文字识别和显式目标定位，但难以处理“左边是边牧右边是金毛”“图片是否符合某种风格”等开放视觉语义问题。LLaVA、InternVL 等开源模型具备本地部署潜力，但对硬件要求较高，毕业设计环境下部署和调优成本较大。GPT-4V/GPT-4o 等商业模型能力较强，但调用成本和接口环境具有不确定性。

系统选择 OpenAI-compatible/Qwen 兼容的 VLM 调用方式，主要原因是接口形式统一，既可以接入 Qwen-VL/Qwen-Omni 系列模型，也可以在需要时替换为其他 OpenAI 兼容视觉模型。系统不把 VLM 放在大规模召回阶段，而是放在最终少量候选的排序、空间约束过滤和回答生成阶段，从而平衡效果、成本和延迟。

### 2.4.8 意图识别与查询规划方案选择

意图识别可选方案包括纯启发式规则、Rasa 分类器、多次 LLM 调用和融合式 LLM Query Planner。

纯启发式规则速度快、可解释，但对复杂自然语言和隐含约束适应能力弱。Rasa 适合训练固定 intent 分类器，但在检索改写、实体抽取和复杂图片约束解析方面能力有限。多次 LLM 调用可以先判断 intent，再做场景解析，但会增加延迟和失败点。

本项目采用融合式 LLM Query Planner 作为优先方案。Planner 在一次 JSON 输出中同时返回 `intent`、`confidence`、`search_rewrite`、`entities` 以及通用问答或文搜图约束。若 Planner 不可用或置信度低，系统再回退到 Rasa 和启发式规则。该设计减少了 LLM 调用次数，并把检索 query 的所有权前移到规划层，避免 dispatcher 再次对 LLM 已生成的检索表达进行不必要改写。

### 2.4.9 存储与记忆组件选择

会话记忆和偏好存储可选方案包括仅内存、Redis、MySQL 和混合存储。仅内存实现简单，适合开发测试，但服务重启后数据丢失。Redis 读写快，适合保存近期会话和 pending 澄清状态。MySQL 持久化能力强，适合保存长期用户偏好和历史记录，但操作成本高于内存和 Redis。

系统当前支持 memory、redis、mysql 和 hybrid 多种模式。默认内存模式降低本地运行门槛；当需要持久化或跨进程共享时，可以切换到 Redis 或 MySQL；hybrid 模式则兼顾 Redis 的速度和 MySQL 的持久性。这种设计符合原型系统逐步扩展的需要。

### 2.4.10 前端与测试工具选择

前端选择原生 HTML、CSS 和 JavaScript，而没有采用 React、Vue 等框架。主要原因是毕业设计阶段前端的职责是验证和展示系统能力，包括输入 query、选择意图、填写 URL、展示回答、证据、图片、进度和 runtime flags。原生前端依赖少、部署简单，能够直接由 FastAPI 静态托管，降低系统整体复杂度。

测试工具选择 pytest。系统测试重点是 API 契约、请求边界、URL 安全、图片代理、parser cache、RAG fallback、dispatcher URL 去重和 planner 路径等关键逻辑。pytest 与 FastAPI TestClient 集成方便，适合本项目进行轻量回归测试。

## 2.5 关键技术分析

### 2.5.1 检索增强生成技术

检索增强生成的基本思想是在生成模型回答之前引入外部知识检索，使模型回答能够基于可追溯证据。传统 RAG 流程通常包括文档采集、清洗、切分、向量化、入库、查询向量化、相关片段检索、上下文构造和答案生成。本系统在通用问答中采用搜索召回、网页采集、BGE 重排和 RAGAnything 查询的组合，使回答不再完全依赖模型参数知识，而是尽量建立在网页证据之上。

### 2.5.2 多模态 RAG 技术

多模态 RAG 将检索对象从纯文本扩展到图片、表格、公式和网页结构等异构内容。其关键问题在于如何把不同模态转换为统一知识表示。本系统通过 `SourceDoc`、`ModalElement` 和 `NormalizedDocument` 等结构承接网页和图片证据，再通过 RAGAnything Bridge 转换为多模态 RAG 引擎可处理的 `content_list`。这种方案使网页中的正文、图片和表格能够以统一证据对象进入后续检索与生成流程。

### 2.5.3 网页解析与结构化转换技术

网页解析是通用问答链路的上游基础。现代网页常包含 JavaScript 动态内容、导航栏、广告、图片、表格和多媒体资源。直接抓取 HTML 并输入模型容易引入大量噪声。系统基于 Crawl4AI 的网页采集能力获取浏览器渲染后的页面内容，并保留 Markdown、HTML、media、links、tables 和 metadata 等结构。Markdown 相比原始 HTML 更适合大模型处理，表格和图片则作为多模态元素继续保留，为后续多模态 RAG 提供输入。

### 2.5.4 跨模态图文匹配技术

文搜图任务需要判断文本描述与图片内容是否匹配。CLIP 类模型通过图文对比学习把文本和图片映射到同一语义空间，适合进行候选图片粗排。对于中文用户 query，Chinese-CLIP 更适合捕捉中文主体、场景和属性表达。系统先用 Chinese-CLIP 缩小候选范围，再用 VLM 对小规模候选进行精细判断，从而提高检索质量并控制推理成本。

### 2.5.5 视觉语言模型推理技术

VLM 能够同时接收图像和文本，适合完成图片描述、视觉问答、空间关系判断和结果解释。本系统在文搜图最终回答阶段使用 VLM，对候选图片进行排序、选择和自然语言解释。对于“左边是 A，右边是 B”等硬约束，系统会额外触发严格空间过滤，要求 VLM 判断候选是否真正满足位置关系。如果没有严格满足条件的图片，系统应明确说明结果不足，而不是仅返回同时包含两个主体但位置错误的图片。

### 2.5.6 LLM 结构化规划技术

系统使用 LLM Query Planner 将自然语言请求转换为可执行计划。Planner 的输出不是最终答案，而是结构化 JSON，包括意图、置信度、检索改写、实体和约束。该技术的价值在于把自然语言理解结果显式化，使后续 dispatcher、RAG 和图片 pipeline 不需要重复猜测用户意图。同时，结构化 JSON 也便于校验、缓存和 fallback。

### 2.5.7 安全访问与降级技术

系统涉及用户输入 URL、搜索结果 URL、图片 URL 和本地缓存路径，存在 SSRF 与任意文件读取风险。因此系统在请求模型、dispatcher 和图片代理处设置多层 URL 安全校验，拒绝非 http/https、localhost、私有 IP、link-local 和 reserved 地址。图片代理在每次重定向后重新校验目标地址，本地图片读取也限制在允许目录内。

降级技术用于保证外部依赖不稳定时系统仍可运行。搜索、Crawl4AI、BGE、RAGAnything、图片 pipeline 和 VLM 都可能失败，系统通过 runtime flags 标记 fallback 路径，并在允许占位降级时返回可解释结果。这样既保证原型系统可演示，也能让测试和调试人员明确知道当前结果是否来自完整链路。

## 2.6 基本方案制定

综合需求分析、可行性分析和组件选型，本系统确定采用“统一编排服务 + 双任务链路 + 多模态证据适配 + 安全可观测支撑”的总体方案。

系统以 FastAPI 主服务作为统一入口。用户通过 Web 前端或 API 提交请求后，系统首先进行 Pydantic 参数校验、request_id 初始化、runtime flags 重置和 progress 任务创建。若用户没有显式指定 intent，系统优先调用 LLM Query Planner 进行意图识别和检索规划；当 Planner 不可用或低置信度时，回退到 Rasa 和启发式规则。若存在 pending 澄清状态，系统会先把用户补充内容与原始问题合并，再重新进入规划流程。

在通用问答链路中，系统采用“搜索召回—摘要重排—安全过滤—网页采集—正文重排—RAG 入库—RAG 查询”的方案。用户提供 `source_docs` 时直接作为证据；用户提供 URL 时直接采集指定网页；用户只提供开放问题时，系统通过搜索服务召回候选网页，用 BGE 对标题和摘要进行重排，过滤不安全 URL 后并发抓取网页正文，再根据配置执行正文级重排。最终，网页和文档证据被转换为 `NormalizedDocument`，通过 RAGAnything Bridge 入库并查询生成答案。

在文搜图链路中，系统采用“约束解析—图片召回—URL 去重—可达性验证—本地缓存—Chinese-CLIP 粗排—VLM 最终回答”的方案。Planner 或 fallback parser 生成 `image_search_query` 和结构化图片约束，dispatcher 使用该检索 query 调用图像 pipeline。pipeline 返回的图片被封装为 `ModalElement` 和 `SourceDoc`。文搜图默认跳过普通 RAG 入库，因为该任务的核心证据是图片本身，最终由 VLM 直接基于图片候选完成排序、空间约束过滤和回答生成。

系统支撑层包括会话记忆、安全过滤、进度事件、指标统计和运行标记。MemoryClient 保存历史上下文、用户偏好和 pending 澄清状态；URL safety 模块统一处理公开 URL 判断；progress 事件用于前端展示执行阶段；metrics 用于统计总体请求和 fallback 趋势；runtime flags 用于在单次响应中暴露实际执行路径。

总体方案可以概括为：

```text
用户请求
  -> FastAPI 统一入口
  -> Query Planner / Rasa / Heuristic
  -> Clarification 判断
  -> QueryExecutionContext
  -> TaskDispatcher
      -> general_qa: search + rerank + crawl + RAG
      -> image_search: image retrieval + cache + CLIP + VLM
  -> MinAdapter 归一化
  -> RAGAnything Bridge 或 VLM 回答
  -> QueryResponse(answer, evidence, images, trace_id, latency_ms, route, runtime_flags)
```

该方案的特点包括：第一，统一 API 屏蔽了通用问答和文搜图的内部差异；第二，Planner 将意图判断和检索规划合并，减少重复 LLM 调用；第三，网页证据通过 Crawl4AI 集成和 RAGAnything Bridge 保留多模态结构；第四，图片链路把可访问性验证放在排序前，避免不可用图片污染后续模型调用；第五，安全校验、降级和可观测机制贯穿请求全生命周期。

## 2.7 本章小结

本章对多模态 RAG 原型系统进行了方案论证。首先，从通用问答、网页采集、多模态 RAG、文搜图、澄清对话、前端展示、安全控制和可观测性等方面分析了系统需求；其次，从技术、经济和运行维护角度说明系统具备可行性；随后重点比较了 Python/FastAPI、Pydantic、Crawl4AI、RAGAnything、BGE、Chinese-CLIP、VLM、Rasa、Redis/MySQL 和前端测试工具等开发组件的选择依据；最后确定了以 FastAPI 为统一编排入口、以 Crawl4AI 和 RAGAnything 的开源能力为核心集成对象、以图像 pipeline 和 VLM 支撑文搜图、以安全与可观测机制保障长链路运行的总体技术方案。

通过本章论证可以看出，本课题的核心并不是简单调用某一个模型，而是将网页解析、跨模态检索、RAG 入库、视觉理解和工程可靠性机制组织为统一系统。后续第三章将在本章确定的基本方案基础上，对系统总体架构、数据流转、模块划分和接口设计进行详细说明。
