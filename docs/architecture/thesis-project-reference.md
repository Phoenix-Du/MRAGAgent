> 范围：覆盖本仓库与已接入第三方能力（检索、RAG、VLM、CLIP、爬虫、重排等）。

## 1 绪论

### 1.1 课题背景、目的与意义

本项目要解决的是“统一处理通用问答与文搜图”的工程问题。传统系统往往把文本问答与图像检索拆成两套流程，导致开发成本高、维护复杂、输出风格不一致。项目通过统一 API 入口、统一请求数据结构与统一编排策略，将 `general_qa` 与 `image_search` 两条链路放在同一个服务中运行，并通过结构化约束与多阶段过滤提升图像场景准确率。

工程价值体现在：
- 统一架构：一个入口处理文本与图像场景；
- 高可用：外部服务失败时有明确 fallback；
- 可解释：请求级 runtime flags 标明实际执行路径；
- 可扩展：可替换模型、检索源和桥接服务。

### 1.2 国内外研究与工程现状（可写方向）

- RAG 在文本问答成熟，但多模态证据融合仍是工程难点；
- 图搜系统在复杂关系（左右、动作、数量）容易“检索到相关但不满足约束”；
- VLM 能提升语义理解，但若候选池质量差，结果仍不稳定。

本项目不是新模型研究，而是多能力协同的工程创新：意图解析、检索改写、候选过滤、VLM 重排、严格约束过滤、答案生成和可视化回传形成闭环。

### 1.3 论文主要内容与结构

建议按“问题定义 → 方案论证 → 设计 → 实现 → 测试 → 总结”展开，突出本系统的可用性和可解释性设计。

## 2 方案论证

### 2.1 需求分析

功能需求：
- 支持统一问答 API；
- 同时支持通用问答和文搜图；
- 支持复杂图像语义（空间、动作、排除词）；
- 图片返回必须稳定可显示；
- 系统要有错误兜底。

非功能需求：
- 性能可接受（秒级响应）；
- 可观测（指标和请求标记）；
- 易维护（分层清晰，模块可替换）；
- 安全（URL 校验、图片代理约束）。

### 2.2 可行性分析

- 技术可行：FastAPI + Pydantic + httpx + VLM/CLIP 组件成熟；
- 工程可行：模块边界明确，核心链路有自动化测试；
- 风险可控：多级 fallback 与超时重试可应对外部不稳定。

### 2.3 开发工具与技术选型

- Web/API：FastAPI
- 数据建模：Pydantic
- 网络请求：httpx（关键调用禁用环境代理污染）
- 图像召回：SerpAPI
- 图文匹配：Chinese-CLIP
- 多模态理解：Qwen VLM
- RAG 桥接：RAGAnything bridge
- 前端：原生 HTML/CSS/JS

### 2.4 关键技术分析

- 意图解析：LLM schema 主导 + heuristic 兜底；
- 检索词解耦：`query` 与 `image_search_query` 分离；
- 文搜图流水线：召回→可达→CLIP→VLM→严格过滤；
- 统一可观测：metrics + runtime flags。

## 3 系统设计

### 3.1 总体架构设计

系统按四层组织：
- API 编排层：`app/api/chat.py`
- 服务与算法层：`app/services/*`
- 集成桥接层：`app/integrations/*`
- 展示层：`web/*`

主链路：
1. 接收请求并归一化；
2. 决策意图（general_qa/image_search）；
3. 执行对应链路；
4. 统一输出答案、证据、图片、运行标志。

### 3.2 核心数据结构设计

关键模型位于 `app/models/schemas.py`：
- `QueryRequest`：包含原始 query、图搜 query、intent、约束等；
- `ImageSearchConstraints`：subjects、synonyms、spatial_relations、action_relations、exclude_terms；
- `QueryResponse`：answer、images、evidence、runtime_flags。

关键演化点：
- `image_search_query` 新增后，检索与回答彻底解耦；
- `local_path` 被强化为前端稳定展示的核心字段；
- `runtime_flags` 用于请求路径解释和调试。

### 3.3 功能模块设计

- `chat.py`：流程控制与策略路由；
- `dispatcher.py`：搜索、爬虫、重排与材料准备；
- `image_query_parser.py`：图搜约束解析；
- `image_pipeline_bridge.py`：图像检索及排序；
- `image_search_vlm_answer.py`：最终图像回答与约束过滤；
- `rag_client.py` / `raganything_bridge.py`：RAG 接入与降级；
- `qwen_vlm_images.py`：图片可达与缓存辅助能力。

## 4 系统实现

### 4.1 统一请求处理实现

入口 `/v1/chat/query`：
- 校验请求；
- 判定意图；
- 触发对应流程；
- 汇总回答并附带运行标记。

意图策略：
- 显式 intent 优先；
- 否则使用解析模型；
- 失败时回退 heuristic。

### 4.2 结构化意图解析实现

图搜解析位于 `app/services/image_query_parser.py`，采用：
- LLM 输出强约束 JSON；
- JSON 容错清洗（代码块和尾逗号）；
- 缓存（TTL）与重试；
- 失败自动回退 heuristic。

该模块解析结果直接驱动：
- 搜索 query 改写；
- 空间/动作约束；
- VLM 严格过滤。

### 4.3 文搜图流水线实现（核心）

实现文件：`app/integrations/image_pipeline_bridge.py`

阶段化处理：
1. **召回**：SerpAPI 拉取候选（支持多 query 变体）；
2. **可访问性过滤**：并发探测并缓存本地路径；
3. **CLIP 粗排**：保留较大池（默认可到 `top_k*4`）；
4. **VLM 精排**：结合查询语义排序；
5. **严格空间过滤**：在扩大候选池执行，避免 top5 误杀。

已落地优化：
- 双 query 检索并去重；
- CLIP 保留池扩大；
- strict spatial filtering 池扩大到 `max(top_k*2, 10)`。

### 4.4 图像回答实现

`app/services/image_search_vlm_answer.py`：
- 对 VLM 排名候选执行严格空间关系验证；
- 若过滤结果变化，重新生成答案，确保“回答与展示图一致”；
- 返回前再次确保 `local_path` 可用，减少前端红叉。

### 4.5 通用问答实现

`app/services/dispatcher.py` 与 `app/services/rag_client.py`：
- 搜索候选→正文抓取→重排→RAG ingest/query；
- 远程 RAG 失败时本地 fallback；
- 标记 `rag_ingest_fallback`、`rag_query_fallback` 便于诊断。

### 4.6 图片代理与前端渲染实现

后端图片代理位于 `chat.py` 的 `/v1/chat/image-proxy`：
- 防止前端直接拉外链失败；
- 校验本地路径与 URL；
- 统一返回可展示图片流。

前端位于 `web/static/app.js`：
- 优先 `local_path` 渲染；
- 展示 runtime flags；
- 与后端编排结果保持一致。

### 4.7 配置与部署实现

- 主配置：`app/core/settings.py`
- bridge 配置：`app/integrations/bridge_settings.py`
- 环境变量样例：`.env.example`

典型可调参数：
- `image_pipeline_timeout_seconds`
- `parser_read_timeout_seconds`
- `rag_ingest_timeout_seconds`
- `IMAGE_MULTI_QUERY_ENABLED`
- `IMAGE_MULTI_QUERY_MAX_VARIANTS`

## 附录 D：可直接扩写的长文素材（按章节拆分）

### D.1 绪论扩写素材

随着多模态大模型能力提升，用户对智能系统的期待已经从“只回答文字问题”转向“能够理解文字、图片、网页、场景关系的综合助手”。然而在真实工程中，系统通常是“文本问答一套、图像检索一套、前端展示再一套”，每条链路分别优化，最终造成以下问题：第一，用户输入同样一句自然语言，系统在不同模式下给出的理解逻辑不一致；第二，当结果错误时，开发者无法快速定位是意图识别错、召回错、重排错，还是渲染错；第三，当某个第三方服务波动或超时时，系统容易整体失效，缺少稳态降级策略。

本项目的核心目标不是单点提升某个模型指标，而是构建一套“可运行、可维护、可解释、可扩展”的多模态问答工程体系。系统从一开始就把第三方能力看作整体架构的一部分：搜索引擎负责广覆盖召回，CLIP 负责快速语义筛选，VLM 负责精细语义判断与回答，RAG bridge 负责跨知识源统一接入。通过统一入口编排，系统把“模型能力”转化为“产品能力”，并通过 runtime flags 与指标统计把黑盒链路变成可观测链路。

从研究意义上看，该项目适合作为应用型毕业设计：它不依赖大规模自训练数据，也不要求改造底层模型，而是通过工程系统设计解决真实业务痛点。这类工作与“工业界 AI 应用落地”的能力模型高度一致，强调需求分析、架构拆分、容错策略、实验验证和持续迭代。论文可以重点展示：如何把多个不稳定外部能力编排成稳定系统；如何把语义约束贯穿到检索和回答；如何通过可观测机制支持问题定位与迭代优化。

### D.2 方案论证扩写素材

#### D.2.1 为什么采用“统一入口 + 分层服务”

若按传统方式开发，通用问答与文搜图会形成两套接口，两套状态管理，两套日志体系。短期看开发快，长期会出现逻辑分叉：同样“左边是A右边是B”的语义在图搜链路被处理，在通用问答链路被忽略，用户体验断裂。统一入口方案可以把意图识别、约束提取、参数校验、审计日志、风险控制放在同一层，减少重复实现和策略漂移。

分层服务的优势在于“高内聚、低耦合”：
- API 层只做编排，不绑定具体模型；
- 服务层负责算法策略，可替换不同模型供应商；
- 集成层负责第三方协议差异；
- 前端层只消费标准响应结构。

这种设计让项目在后续切换 VLM、增加检索源、替换 RAG 后端时，修改范围可控。

#### D.2.2 为什么采用“LLM 解析 + heuristic 兜底”

纯规则解析在固定句式下成本低，但面对自然语言长尾会迅速失效；纯 LLM 解析在效果上更强，但会受网络波动、限流、输出格式不稳定影响。组合策略的工程价值在于：当 LLM 稳定时获得高语义覆盖，当 LLM 不可用时依然可以服务最常见场景。  
该项目中，LLM 负责输出结构化约束（subjects、spatial、actions、rewrite），heuristic 只保底主谓宾和方位词抽取。这种职责切分避免了“在 Python 中不断堆词表和正则”的维护灾难。

#### D.2.3 为什么采用“多阶段图像过滤”

搜索引擎召回强调覆盖，噪声不可避免。若直接交给 VLM 全量判断，成本高、时延大。项目采用“可达性过滤 → CLIP 粗排 → VLM 精排 → 严格空间过滤”的分层策略，背后逻辑是：先用廉价算子去除明显无效候选，再用昂贵算子做高价值决策。  
这种流水线不仅提高准确率，也更符合工程性能预算。

### D.3 系统设计扩写素材

#### D.3.1 全链路时序设计（文字版）

1) 用户在前端提交 query；  
2) 后端创建 request 上下文并初始化 runtime flags；  
3) 执行意图决策（显式 intent / LLM / Rasa / heuristic）；  
4) 若是 general_qa，进入搜索与 crawl 流程，构造文档输入并请求 RAG；  
5) 若是 image_search，先解析结构化约束并生成 image_search_query；  
6) 图像 pipeline 检索候选并阶段化过滤；  
7) VLM 依据用户原 query 进行排序与回答；  
8) 若存在空间约束，执行 strict filter 并必要时重答；  
9) 统一整理 response（answer、images、evidence、runtime_flags）；  
10) 前端渲染答案与图片，图片通过后端代理获取。

#### D.3.2 关键字段设计动机

- `query`：保留用户原语义，保证回答阶段“忠于用户问题”；  
- `image_search_query`：检索优化语句，仅用于召回；  
- `subject_synonyms`：由 LLM 输出同义扩展，提升召回覆盖；  
- `action_relations`：保留原动作关系，避免仅靠名词召回；  
- `local_path`：解决前端外链失效问题，提升可展示稳定性；  
- `runtime_flags`：记录链路决策，为调试和实验提供依据。

#### D.3.3 约束传播设计

项目的关键不是“提取约束”，而是“让约束真正影响结果”：
- 在 query rewrite 中使用主体和动作；
- 在 VLM strict filter 中使用空间约束；
- 在回答重生成阶段确保图文一致；
- 在 response 中附带执行痕迹，让约束传播可见化。

### D.4 系统实现扩写素材

#### D.4.1 API 编排实现细节

`chat.py` 中的处理逻辑可分为六步：
- 入参校验：保证 query、max_images、source_docs 等结构合法；
- 意图确定：融合显式输入、解析结果和兜底规则；
- 澄清判定：当约束冲突或信息不足时触发 clarification；
- 路由执行：分派到 general_qa 或 image_search；
- 统一组装：保证 response 字段稳定；
- 运行标志：记录策略选择、fallback、过滤行为。

这种实现方式保证接口契约稳定，即使内部模型更换，前端与调用方不需要重写协议。

#### D.4.2 general_qa 实现细节

通用问答路径强调“证据先行”：
- 先取搜索候选，再抓取正文；
- 通过摘要级与正文级重排减少噪声；
- 将整理后的内容交由 RAG bridge ingest/query；
- 若远程失败，启用本地 fallback，保证可回答。

此路径可在论文中强调“证据链”概念：答案不是直接幻想生成，而是建立在可追踪来源上。

#### D.4.3 image_search 实现细节

文搜图路径可总结为“两次语义对齐”：
- 第一次语义对齐：query rewrite + CLIP，把检索结果拉向用户目标；
- 第二次语义对齐：VLM 排序 + strict spatial filter，把结果拉向结构化约束。

工程上最容易忽视的是“可展示性”：即使语义对齐正确，若图片无法加载，用户仍认为系统失败。因此本项目把 `local_path` 强化为结果交付标准的一部分，确保最终展示稳定。

#### D.4.4 第三方集成作为系统一部分

论文可以把第三方能力写成“系统外部子系统”而非“调用黑盒”：
- SerpAPI：外部图像召回子系统；
- DashScope/Qwen：多模态语义决策子系统；
- RAGAnything：知识接入与问答子系统；
- Crawl4AI：网页结构化采集子系统；
- BGE reranker：文本相关性优化子系统。

写作时强调：项目通过协议统一、超时重试、结果标准化把这些外部系统纳入本项目控制平面，实现了“松耦合的统一系统”。

### D.5 测试与分析扩写素材

#### D.5.1 测试设计原则

- 功能正确性优先：先确认链路可用，再优化性能；
- 关键路径覆盖：意图路由、约束传播、图片展示；
- 异常路径覆盖：超时、空结果、模型失败、网络错误；
- 回归测试优先：每次关键改动后执行完整测试。

#### D.5.2 建议的实验分组

可设计四组对照：
- A 组：单 query + 小池过滤；
- B 组：多 query + 小池过滤；
- C 组：单 query + 大池 strict；
- D 组：多 query + 大池 strict（当前推荐）。

对每组记录：
- raw 候选数；
- 可达候选数；
- CLIP 保留数；
- VLM 重排后候选；
- strict 命中数；
- 最终展示数；
- 用户约束满足率。

#### D.5.3 典型分析结论模板

“实验结果显示，多 query 召回显著提升了 strict 阶段候选覆盖，CLIP 保留池扩大后 VLM 可选空间增加，最终空间关系满足率上升。虽然平均时延略增，但在可接受范围内。综合准确率与可用性，D 组配置表现最优。”

#### D.5.4 指标解释模板

- 若 fallback 率高：说明上游依赖稳定性不足或超时配置偏紧；
- 若 strict 命中率低：说明召回 query 质量不足或 CLIP 截断过早；
- 若可展示率低：说明缓存回填、代理策略或图片源质量有问题；
- 若回答一致性低：说明 strict 后未重答或 prompt 对齐不足。

### D.6 局限与优化扩写素材

#### D.6.1 局限性

1. 对外部检索源依赖较强，若源质量差会传导至后续阶段；  
2. 复杂空间语义仍可能出现歧义，尤其是遮挡、镜像、拼图场景；  
3. 当前评测集规模有限，缺少更大规模标注集支持统计显著性；  
4. 某些 fallback 仍偏工程启发式，尚未形成学习型决策。  

#### D.6.2 优化方向

1. 引入 query 变体自适应权重；  
2. 将 strict hard filter 升级为 soft score 融合；  
3. 构建离线 benchmark，持续评估空间关系与动作关系；  
4. 建立阶段化 tracing，降低复杂问题定位成本；  
5. 增加对图像来源质量的先验评估，减少低质量源占比。

## 附录 F：论文可复用段落（示例）

### F.1 “系统价值”段落示例

本系统通过统一编排层实现通用问答与文搜图一体化处理，避免了传统多系统并行开发造成的逻辑分叉和维护负担。相比单一模型驱动方案，系统强调多阶段协同与工程鲁棒性：在检索阶段通过 query 变体与并发可达性过滤提升召回质量，在筛选阶段通过 CLIP 与 VLM 逐级提纯候选，在回答阶段通过严格空间约束过滤保证图文一致，在交付阶段通过本地缓存路径与后端代理提升图片可展示稳定性。该设计将第三方能力纳入统一控制平面，使系统在外部依赖波动场景下仍能保持可用输出。

### F.2 “实验结论”段落示例

实验结果表明，采用多 query 召回与扩大 strict 过滤池后，空间关系约束命中率显著提高，最终结果与用户意图的一致性提升明显。虽然系统平均响应时间较基础方案有所增加，但通过并发优化与阶段化超时控制，整体时延仍满足交互式应用要求。结合 fallback 统计可以看出，系统在外部服务异常情况下仍可返回可用结果，验证了所设计容错机制的有效性。

### F.3 “不足与展望”段落示例

目前系统在复杂视觉场景下仍存在误判风险，主要来源于外部检索源噪声与视觉模型对细粒度空间关系的理解偏差。后续将引入更系统化的离线评测数据集，探索 strict 过滤的软评分融合策略，并完善全链路 tracing 与可观测体系，以进一步提升系统准确率、稳定性与可维护性。
